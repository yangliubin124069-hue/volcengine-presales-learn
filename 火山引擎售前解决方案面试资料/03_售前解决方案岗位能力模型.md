# 售前解决方案岗位能力模型

## 一、岗位本质

售前解决方案岗位的本质是“技术商业化翻译器”和“成交前的技术负责人”。

它连接四类人：

- 客户业务方：关心业务目标、效率、体验、风险和预算。
- 客户技术方：关心架构、性能、稳定性、安全、兼容、迁移成本。
- 内部销售：关心商机推进、客户关系、预算、采购流程、竞争策略。
- 内部产研交付：关心需求边界、产品能力、交付可行性、资源投入。

你的价值在于把客户的模糊需求转成能落地、能验证、能采购、能交付的解决方案。

## 二、典型职责拆解

### 1. 售前交流

目标：建立信任，发现客户真实问题。

你要做：

- 了解客户行业、业务模式、组织结构和数字化阶段。
- 问清楚业务目标、现状架构、历史问题、预算和时间约束。
- 区分“客户说的需求”和“客户真正要解决的问题”。
- 用行业案例和技术方案引导客户形成共识。

好答案示例：

我会先把交流分成业务、技术、项目三条线。业务上确认目标指标，比如降本、增效、增长、体验提升。技术上确认现有架构、数据源、并发、性能、安全合规和系统依赖。项目上确认预算、决策链、时间节点、POC 成功标准和交付边界。这样可以避免一开始就陷入产品功能介绍。

### 2. 需求分析

目标：把“想要一个 AI 客服/数据中台/云原生平台”拆成具体需求。

常用维度：

- 业务需求：要解决什么业务问题，谁使用，使用频率，成功指标。
- 功能需求：需要哪些功能，必须项和可选项是什么。
- 非功能需求：性能、可用性、扩展性、兼容性、安全、合规。
- 数据需求：数据来源、质量、权限、更新频率、历史数据迁移。
- 集成需求：和哪些系统对接，接口协议，身份认证，单点登录。
- 运维需求：监控、告警、审计、备份、容灾、发布流程。
- 商务需求：预算、采购方式、部署形态、合同边界。

### 3. 方案设计

目标：形成客户认可的端到端方案。

方案文档建议包含：

- 背景与目标
- 现状与痛点
- 需求范围
- 总体架构
- 产品选型和模块说明
- 数据流和调用链路
- 安全与合规设计
- 高可用与容灾设计
- 部署方案
- POC 计划
- 实施里程碑
- 风险与应对
- 成本估算
- 客户收益

方案设计要体现取舍：

- 为什么用公有云、私有化或混合云。
- 为什么选某个模型、数据库、存储或计算形态。
- 哪些能力一期做，哪些二期做。
- 哪些需求需要定制开发，哪些用标准产品。

### 4. POC 设计与主导

目标：用最短路径证明方案可行，并推动客户进入采购。

POC 不是“做个 demo”。POC 必须有：

- 范围：验证哪些能力，不验证哪些能力。
- 数据：使用真实数据、脱敏数据还是样例数据。
- 指标：准确率、延迟、吞吐、稳定性、成本、安全等。
- 环境：公有云、客户内网、专线、测试账号、权限。
- 分工：客户、销售、售前、产品、研发、交付各自负责什么。
- 时间表：准备、部署、调试、测试、汇报。
- 成功标准：达到什么阈值算通过。
- 风险预案：数据不到位、接口不通、性能不达标怎么办。

### 5. 投标与讲标

目标：把技术方案转成采购语言和评分项。

讲标结构：

1. 对客户业务和项目目标的理解。
2. 当前痛点和建设必要性。
3. 方案总体思路。
4. 核心架构和关键能力。
5. 与招标要求的逐项响应。
6. 实施计划和项目保障。
7. 安全、合规、高可用、运维保障。
8. 成功案例和预期收益。

注意：

- 招标场景不要过度炫技，要贴评分项。
- 每个技术亮点都要对应客户价值。
- 对无法满足的条款要提前澄清，不要临场硬承诺。

### 6. 产品反馈与内部协同

售前经常是产品和市场的一线传感器。

你要能把客户需求分级：

- 标准能力：已有产品能满足。
- 配置能力：通过参数、模板、插件、流程配置满足。
- 轻量定制：少量开发或集成可满足。
- 产品缺口：需要产研评估 roadmap。
- 不建议承接：成本高、交付风险大、商业价值不足。

向产研反馈时，不要只说“客户要某功能”，要说：

- 客户是谁，行业和规模如何。
- 业务场景是什么。
- 该需求影响商机金额和紧急程度如何。
- 竞品是否已经支持。
- 是否可抽象成通用能力。
- 临时 workaround 是什么。

## 三、能力模型

### 1. 技术深度

至少要能讲清：

- 云计算基础：计算、网络、存储、数据库、负载均衡、容灾。
- 云原生：Docker、Kubernetes、微服务、DevOps、灰度发布。
- 数据：数据集成、数仓、湖仓、实时计算、数据治理、BI。
- AI：大模型调用、RAG、向量数据库、模型评测、精调、Agent。
- 安全：身份认证、权限、加密、审计、数据脱敏、等保合规。

### 2. 架构能力

能画出并解释：

- 三层应用架构
- 微服务架构
- 数据中台架构
- 湖仓一体架构
- RAG 知识库架构
- AI Agent 应用架构
- 混合云/多云连接架构
- 高可用与容灾架构

### 3. 业务理解

不同行业关注点不同：

- 金融：合规、风控、稳定性、数据安全、审计、私有化/混合云。
- 零售：会员运营、营销转化、供应链、私域、公域投放、数据洞察。
- 汽车：智能座舱、车联网、内容生态、数据闭环、低延迟体验。
- 泛互联网：高并发、成本、增长、推荐、音视频、全球/多地域。
- 科研医药：数据管理、算力调度、模型训练、文献助手、合规。

### 4. 商业推进

售前要懂商机阶段：

线索 -> 初次交流 -> 需求调研 -> 方案汇报 -> POC -> 商务报价 -> 投标/采购 -> 合同 -> 交付

每个阶段要回答：

- 当前客户是否有明确痛点。
- 是否有预算和时间节点。
- 决策人、影响人、使用人是谁。
- 竞争对手是谁。
- 下一步动作是什么。
- 阻塞点是什么。

### 5. 表达与文档

售前输出常见材料：

- 客户交流 PPT
- 技术方案书
- POC 测试方案
- POC 测试报告
- 投标技术应答
- 架构图
- 会议纪要
- SOW 工作说明书
- 客户案例和行业方案沉淀

## 四、面试中建议展示的特质

- 结构化：回答先框架后细节。
- 客户感：不只讲技术，还讲客户目标。
- 边界感：知道什么能承诺，什么要评估。
- 推动力：能组织 POC、协调产研交付、推动下一步。
- 学习快：云厂商产品多，能快速补齐产品和行业知识。
- 稳定可信：售前面对客户，表达要准确、可靠、不过度承诺。

## 五、常用回答框架

### 框架 1：需求调研

业务目标 -> 用户角色 -> 当前架构 -> 痛点优先级 -> 数据与系统依赖 -> 非功能指标 -> 预算时间 -> 决策链 -> POC 标准

### 框架 2：方案设计

场景理解 -> 总体架构 -> 产品组合 -> 核心流程 -> 安全合规 -> 高可用运维 -> 实施计划 -> 成本收益 -> 风险预案

### 框架 3：客户异议

先确认顾虑 -> 拆成事实问题和感受问题 -> 给证据或验证路径 -> 给替代方案 -> 明确下一步

### 框架 4：POC 汇报

目标 -> 环境 -> 数据 -> 测试项 -> 结果 -> 未满足项 -> 原因分析 -> 优化建议 -> 商务推进建议
