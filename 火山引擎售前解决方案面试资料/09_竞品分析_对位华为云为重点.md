## 一、为什么要懂竞品 —— 竞品视角的四个层次

售前的"竞品视角"不是参数对照表，而是 **四件事的合体**：

| 层次 | 能力 | 关键问题 |
|---|---|---|
| **L1 客户画像** | 知道每家厂商在什么客户群里强 | 这个客户是不是友商主场？ |
| **L2 真实优劣势** | 知道每家真正能做和做不到的事 | 客户说"XX 也能做"时怎么回应？ |
| **L3 对位话术** | 不贬低、不撒谎、承接 + 破局 | 客户对我的信任怎么建立？ |
| **L4 维度迁移** | 把"功能比对"挪到"业务价值 / TCO / 服务能力" | 我赢单的真正抓手在哪？ |

> 一句话总结：**最差的售前贬低对手，好的售前承认事实并扭转维度，最强的售前能给客户拿一个对手做不到的 POC**。

### 3.2 华为云的真实优势（要先承认）

售前面试如果让你"评价华为云"，绝不能只说短板。先承认下面这些：

| 优势 | 具体表现 | 别企图否认 |
|---|---|---|
| **政企渠道与关系** | 华为系销售 + 解决方案架构师在政企客户根基极深 | 这是硬实力 |
| **全栈自主可控** | 鲲鹏（CPU）+ 昇腾（AI）+ 欧拉（OS）+ 高斯（DB）+ 鸿蒙 + HCS | 信创单子里几乎默选 |
| **行业方案深度** | 数字政府、公共安全、智慧应急、烟草、电网、煤矿等垂直方案产品化程度高 | 现成解决方案 + 大量参考案例 |
| **HCS 私有云成熟度** | 国内私有云份额第一，金融/政府大量主流场景 | 私有云对位绕不开 |
| **运营商联合方案** | 与三大运营商在政务云、行业云上的深度合作 | 央国企单子里默认对手 |

### 3.3 华为云的真实短板（火山引擎可以打的位置）

> 这些不是"贬低"，是行业内售前圈共识。客户真正问起来时，提供"可验证路径"比说大话有用。

| 短板 | 表现 | 火山引擎对应优势 |
|---|---|---|
| **大模型迭代慢、闭环差** | 盘古更新周期长，应用市场不开放，开发者生态弱 | 豆包月级迭代，火山方舟开放 + 抖音/电商真实场景验证 |
| **AI 应用层产品化弱** | 大模型 → 应用之间的工程链路（RAG、Agent、知识库）做得偏 toG，对互联网客户慢 | 火山方舟 + VikingDB + Agent 链路对开发者友好 |
| **大数据栈相对老派** | FusionInsight 偏传统 Hadoop 生态，分析时延、湖仓一体能力对比有差距 | ByteHouse + EMR + LAS 湖仓一体，性能可量化 |
| **互联网/内容客户经验不足** | 短视频、直播、推荐、增长这套打法不是华为强项 | 字节业务原生场景，技术从生产环境长出来 |
| **价格刚性 / 折扣空间收窄** | 央国企单子价格刚性、商务谈判空间小 | 互联网客户能给到更灵活的资源包/账期 |
| **应用市场与开发者社区弱** | 华为云开发者社区活跃度对比有差距 | 火山引擎依托字节技术开放生态 |
| **多租户/SaaS 化能力** | 部分场景 SaaS 化路径不如互联网厂商成熟 | 字节抖店/巨量等 SaaS 平台积累深 |

### 3.4 五个典型对位场景的话术框架

#### 场景 1：客户说"我们已经在用华为云，为什么还要试火山？"

❌ 错误回答："华为云不行，我们更快。"
✅ 正确回答框架：

> "华为云在 [客户具体场景，如政务底座/信创基础设施] 这块做得确实很扎实，咱们没必要换底座。
>
> 我建议是**双云互补**：底座留华为，AI 应用层和数据应用层引入火山。原因是：
>
> 1. **模型选择更灵活**：火山方舟接入了豆包、DeepSeek 等多家模型，客户根据场景选最合适的，而不是绑定一家。
> 2. **AI 应用工程化更成熟**：RAG / Agent / 多模态在火山方舟里是产品化的，从 Demo 到生产几周搞定。
> 3. **场景已被验证**：抖音电商、巨量引擎等是大流量真实场景，技术不是 PPT 出来的。
>
> 我们可以先选 [一个具体场景，如智能客服/合同抽取/会议纪要] 做 1 个月 POC，KPI 定 [具体指标]，验证完不行也不影响您的华为底座。"

**核心：不抢底座，抢应用层；引导双云。**

#### 场景 2：客户说"华为有盘古大模型，你们豆包好在哪？"

> "盘古在 [行业大模型 / 科学计算 / 气象等垂直场景] 上确实有积累，这是华为长期投入的优势。
>
> 豆包的差异点是 **三个 R**：
>
> - **Recent**（迭代更快）—— 月级版本，能跟上业务变化。
> - **Real**（真实场景验证）—— 抖音、电商、客服每天数十亿次调用打磨出来的。
> - **Ready to ship**（应用就绪）—— Pro / Lite / Mini / Code 四档分级 + 火山方舟工程化链路，从模型到应用是一套产品，不是一个 API。
>
> 建议做 [具体业务场景] POC，按 准确率 / 时延 / 成本 / 可解释性 四个指标对比，让数据说话。"

#### 场景 3：客户说"我们要做信创，华为是默选，你们怎么进？"

> "信创基础设施这一层，华为的鲲鹏+欧拉+高斯是确定路径，咱们不强行替换。
>
> 火山引擎的进入点是 **AI 应用与数据智能层**：
>
> - **底座可信创**：豆包大模型支持在国产算力（昇腾/海光）上推理。
> - **数据应用国产化**：DataLeap + ByteHouse 已通过信创认证，能在信创底座上跑分析与中台。
> - **降低绑定风险**：客户多一个国产 AI 厂商选型，避免单源风险。
>
> 我们可以先做 [具体应用场景] 的信创验证 POC，用客户既有信创资源跑，验证完再讨论范围。"

#### 场景 4：客户说"华为方案是端到端打包的，你们呢？"

> "端到端打包确实省心，但代价是**绑定深、扩展慢、决策权在乙方**。
>
> 火山引擎走的是 **'平台 + 开放生态'** 路线：底座、模型、数据、应用四层都开放接口，客户可以选我们的也可以接 ISV 的。
>
> 这意味着：
>
> 1. 您的核心业务系统不用全推倒。
> 2. 后续业务变化时切换成本更低。
> 3. 您的技术团队保留架构话语权。
>
> 我们可以选您**最痛但最容易出价值**的那个环节先做，把价值跑出来再扩展。"

#### 场景 5：客户说"华为有 HCS 私有云，你们专有云能比吗？"

> "HCS 在私有云市场份额第一，这是事实。咱们不去比纯基础设施层。
>
> 火山引擎专有云的差异在 **AI 与数据原生能力**：
>
> - 豆包大模型可私有化部署，模型权重不出域。
> - 火山方舟的 RAG / Agent 链路在专有环境下完整可用。
> - DataLeap / ByteHouse / VikingDB 一并下沉，不只是 IaaS 而是 AI Cloud Stack。
>
> 如果客户场景是**确定的 AI 应用 + 数据合规**，专有云会是更合适的形态；如果场景主要是传统业务系统，HCS 也能跑火山的 AI 模块。两条路都开放讨论。"

### 3.5 政企场景的差异化打法（HCSP-Presales 角度）

如果你跟客户讲数字政府 / 一网通办 / 公共安全 / 智慧应急，对手 90% 是华为。这里给出几个**只有懂华为打法的人才能用上**的破局点：

| 维度 | 华为的打法 | 火山引擎可破局的点 |
|---|---|---|
| **一网通办** | 华为提供整体政务底座 + 统一身份 + 数据共享 | 火山引擎在**政务智能客服 / 政策大模型 / 政民对话** 上更快出效果 |
| **政务大模型** | 盘古政务版 + 行业知识增强 | **政民语言更接地气**（豆包来自 C 端真实对话）+ **更新周期更快** |
| **公共安全** | 视频图像分析 + 统一指挥 + 雪亮工程 | 视频检索 / 多模态 / 实时回放分析在互联网场景验证更彻底 |
| **政务数据中台** | DataArts + FusionInsight 政务版 | DataLeap 在数据治理 + 资产管理上工具链更新更快 |
| **数字员工 / 政务 Agent** | 偏 RPA + 流程自动化 | 火山方舟 Agent + 豆包 + RAG 一站式，落地快 |

> **打华为的核心心法**：不要在底座层硬刚，要在 **"AI 应用价值 + 业务效果"** 这层超越。客户买 IaaS 是买放心，买 AI 是买**结果**。

## 四、阿里云对位（★★★★）

### 4.1 客户画像
- 互联网中大客户、零售品牌、金融科技、政务一网通办（部分省份）
- 资历最深、生态最广、价格最灵活

### 4.2 产品对位重点

| 火山引擎 | 阿里云 | 对位说明 |
|---|---|---|
| 豆包 / 火山方舟 | 通义千问 / 百炼 | 通义生态全 vs 豆包迭代快 + 字节场景 |
| ByteHouse | AnalyticDB / Hologres / MaxCompute | 阿里数仓产品矩阵更细 vs ByteHouse 单点性能强 |
| DataLeap | DataWorks | DataWorks 工具链最成熟，火山追赶但工具链更新更快 |
| VKE | ACK | ACK 资历更老 vs VKE 与字节运维体系打通 |
| TOS / RDS / Redis | OSS / RDS / Redis | 几乎对位 |
| RTC / VOD | RTC / VOD | 阿里有阿里云视频云，火山有抖音/字节技术原生 |
| 增长营销 | QuickAudience / DataPhin | 火山的"增长引擎"+ 抖音生态闭环更强 |

### 4.3 阿里云的真实优势
- 生态最厚（合作伙伴 + ISV + 开发者）
- 数据中台资历最深，DataWorks 工具链最完整
- 价格最灵活，商务空间大
- 通义千问开源生态广，开发者社区活跃

### 4.4 阿里云的短板
- 大模型与应用的整合度不如火山方舟一站式
- 部分老产品技术栈陈旧，迁移成本高
- 互联网巨头身份在部分客户处有"数据安全顾虑"
- 政企单子近两年被华为/天翼挤压

### 4.5 通用话术
> "阿里云的生态是绕不开的优势，咱们不否认。火山引擎的差异在于 **AI 应用工程化更成熟**、**模型选择更开放**（多家模型可选）、**字节场景验证过的真实流量**。建议在 AI 应用 + 增长这两个场景做对比 POC，不动您现有的底座。"

## 六、百度智能云对位（★★）

### 6.1 客户画像
- AI 老品牌客户、智能驾驶、部分政企、智慧交通
- 文心 + AI 语料是历史资产

### 6.2 关键对位
- **文心一言 vs 豆包**：百度做 AI 早，但产品迭代节奏被字节追上；豆包在 ToB 应用化上更快。

- **政务**：百度有部分省份基础，但渠道下沉度不如华为/天翼。

### 6.3 通用话术
> "百度在 AI 行业里是老品牌，文心是中国最早的大模型之一，咱们认。差异在于 **业务场景的真实性**：豆包来自抖音、电商、客服等高频高量场景，反馈循环更短。建议按 [客户业务] 对比指标。"

## 七、AWS / Azure 对位（★★）

### 7.1 国内退场背景
- 2024 年起 AWS / Azure 在中国的合规与服务收缩，多数政企客户已切走
- 主要剩下：出海企业、外企在华、跨国 SaaS

### 7.2 对位核心
- **赢的点**：合规友好、价格本土化、AI 大模型不受出口限制、数据出境合规支持。
- **认输的点**：全球节点、国际生态、外企采购流程默选 AWS。

### 7.3 通用话术
> "出海第一站走 AWS 是合理选择，节点和生态成熟。但**两类场景火山更合适**：① 出海业务回中国分析的数据回流；② 中国客户用 AI 做内容/营销面向海外用户。我们可以做混合架构，AWS 出海 + 火山做 AI 与数据。"

## 八、对位话术四步法（通用框架）

不管面对哪家友商，统一用这套话术：

```
   ① 承认（不贬低）
   ↓
   ② 维度迁移（从功能比对 → 业务价值/TCO/服务）
   ↓
   ③ 给可验证路径（POC + 明确指标）
   ↓
   ④ 留退路（不损害客户既有投资）
```

**原话模板**：

> "[友商名] 在 [具体场景] 做得确实扎实，这是事实。
> 火山引擎的差异不在 [友商强项]，而在 [我方差异点]。
> 建议我们按 [客户业务目标] 做一个 [周期] 的 POC，KPI 定 [指标]，
> 不动您现有的 [既有投资]，验证完了再讨论扩展范围。"

## 九、面试高频题与答题框架

### 9.1 "你怎么看华为云？火山引擎和华为云比有什么差异？"

**答题框架**：
1. 先承认华为云优势（政企渠道、信创自主、HCS 成熟、行业方案多）。
2. 给出火山引擎差异（AI 迭代、应用工程化、互联网场景验证、开放生态）。
3. 推"双云 + 应用层切入"策略。
4. 给一个具体客户场景的 POC 思路。

**示范回答**（30-60 秒）：
> "华为云在政企/信创/私有云这几块的优势我们必须承认，渠道下沉度和行业方案产品化都是头部。火山引擎的差异点是 AI 与数据应用层 —— 豆包月级迭代、火山方舟把 RAG/Agent 做成产品、字节真实场景验证过的技术栈。所以在政企客户里，我们一般不抢底座，从 AI 应用层切入，用 1 个月 POC 把 [客户场景] 的指标拿出来，再讨论扩展。这种打法对客户来说投资风险最小。"

### 9.2 "你打过华为吗？怎么打的？"

**答题框架**：
1. 先讲一个真实项目背景（你的中安/英方经验都能套）。
2. 客户原本默选华为的原因。
3. 你做的工作：调研、对比、POC、价值呈现。
4. 结果：拿下、并存、或丢单的复盘。

**示范回答**（结合你简历）：
> "在 [中安数联] 我打过几个对位华为的政企单子。比如 [柳州政务云灾备] 那个项目，客户最初默选华为的备份方案。我做的工作是：① 详细调研客户实际备份场景，发现华为方案在多租户隔离上有限制；② 用 TeraBackup 设计了三大运营商统一管理方案，做了 1 周 POC；③ 给客户出了一个 TCO 三年对比表。最后我们和华为并存进入项目，承担了多运营商整合那部分。复盘下来，**关键不是技术比拼，是把'客户真实场景的痛点'挖到比对手深一层**。"

### 9.3 "如果客户已经买了通义千问，你怎么推豆包？"

**答题框架**：
1. 不打通义，谈互补。
2. 推"多模型路由"或"按场景选型"。
3. 强调豆包在某个具体场景的差异验证。

**示范回答**：
> "通义千问是好选择，咱们不替换。建议是**多模型并存策略**：火山方舟支持接入多家模型 API，客户按场景选择。比如 [营销文案生成] 用通义，[客服对话] 用豆包，[代码生成] 用 Code 模型。火山方舟做统一调度和成本管控。这样客户的选型更灵活，也避免了模型供应商锁定风险。"

### 9.4 "为什么火山引擎能在政企里跟华为/天翼竞争？"

**答题框架**：
1. 承认我们不是政企赛道的传统选手。
2. 给出我们的破局点：AI 应用 + 字节场景。
3. 案例：豆包政务版、抖音政务号、字节政府客户。
4. 长期判断：AI 浪潮重塑政企采购心智。

**示范回答**：
> "我们不抢华为/天翼的底座位置，我们的破局点在 AI 应用层。三个事实支撑：① 豆包是国内 ToB 调用量最大的模型之一，政务客服、政策问答、文书写作场景成熟；② 抖音政务号生态让我们对'政民对话'语料有独特优势；③ 字节自身是大型央企级别的数字化样本。所以我们的打法不是'替代政务云'，是'让政务云上多一个 AI 选择'。这个角度客户接受度高很多。"

### 9.5 "客户问 '你们 AI 算力供应稳定吗？华为有昇腾'，怎么答？"

**答题框架**：
1. 承认昇腾是华为重要资产，是国产算力主流之一。
2. 火山引擎的算力策略：多元 + 弹性。
3. 给出具体保障措施。
4. 推 POC 而不是空口承诺。

**示范回答**：
> "昇腾是国产 AI 算力的主流之一，华为投入多年。火山引擎的算力策略是 **多元 + 弹性**：NVIDIA H 系列为主，部分场景接入昇腾、海光、天数等国产芯片，根据客户合规和成本要求灵活配置。供应稳定性上，我们背靠字节自有 GPU 集群，业务高峰期能调度抖音/巨量的算力，是真实业务跑过的容量。建议先做 1 个月小批量 POC，按 [推理时延/吞吐/成本/合规] 四个指标实测。"
