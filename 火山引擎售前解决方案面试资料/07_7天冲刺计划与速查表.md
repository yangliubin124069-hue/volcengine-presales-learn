# 7 天冲刺计划与速查表

## 一、7 天冲刺计划

### Day 1：公司和岗位

目标：

- 能用 2 分钟讲清火山引擎是谁。
- 能讲清售前解决方案岗位职责。
- 准备 90 秒自我介绍。

任务：

- 阅读 `01_火山引擎公司业务与面试定位.md`
- 背熟火山引擎关键词：AI 云、字节实践、数据中台、云原生、音视频、行业方案。
- 把自己的经历改写成售前语言。

输出：

- 90 秒自我介绍
- “为什么火山引擎”答案
- “为什么售前解决方案”答案

### Day 2：产品地图

目标：

- 能按场景说产品组合。
- 不要求记参数，但要知道产品在架构中的位置。

任务：

- 阅读 `02_产品技术速览_售前必背.md`
- 重点记 AI、大数据、云原生三条产品线。

输出：

- 企业知识库问答架构
- 数据中台架构
- 云原生迁移架构

### Day 3：行业方案

目标：

- 能对金融、零售、汽车、泛互联网各讲 3 分钟。

任务：

- 阅读 `04_行业场景与方案话术.md`
- 每个行业背：痛点、产品组合、POC 指标。

输出：

- 金融大模型话术
- 零售智能运营话术
- 汽车智能座舱/车联网话术

### Day 4：技术深挖

目标：

- 补齐可能被追问的技术细节。

任务：

- 复习 Kubernetes：Pod、Deployment、Service、Ingress、HPA、ConfigMap、Secret、RBAC。
- 复习数据：Hadoop、Hive、Spark、Flink、Kafka、湖仓、数据治理、血缘、质量。
- 复习 AI：RAG、向量数据库、Embedding、重排、评测、精调、Agent、工具调用。

输出：

- 每个技术点用一句话解释。
- 每个技术点说一个客户价值。

### Day 5：案例和 POC

目标：

- 能现场拆 case。
- 能设计 POC。

任务：

- 阅读 `06_案例演练_POC_讲标.md`
- 每个案例按 8 步框架复述一遍。

输出：

- 零售 AI 导购 POC
- 金融知识助手 POC
- 大数据平台上云 POC

### Day 6：模拟面试

目标：

- 回答自然，不像背稿。

任务：

- 阅读 `05_面试题库与参考回答.md`
- 对每道题录音回答一遍。
- 每题控制在 1 到 3 分钟。

输出：

- 10 道高频题自己的版本
- 3 个 STAR 项目案例

### Day 7：压缩复习

目标：

- 只看高频知识和自己的项目。

任务：

- 看本文件的速查表。
- 看自我介绍。
- 看 3 个 STAR 案例。
- 看反问面试官问题。

不要做：

- 不要临时大范围学新产品。
- 不要背太多参数。
- 不要熬夜。

## 二、面试速查表

### 1. 火山引擎一句话

火山引擎是字节跳动旗下企业级技术服务平台，定位 AI 云，把字节在云计算、数据、AI、音视频、内容和增长中的大规模实践沉淀为企业级产品和行业解决方案。

### 2. 售前岗位一句话

售前解决方案是成交前的技术负责人，负责把客户业务需求转成技术方案、POC、投标材料和可交付边界，并协同销售、产品、研发、交付推动项目落地。

### 3. 火山产品三条主线

AI 主线：

豆包大模型、火山方舟、机器学习平台、VikingDB、知识库、Agent 平台、GPU 云服务器、LAS。

数据主线：

DataSail、DataLeap、EMR、LAS、ByteHouse、Flink、DataWind、TOS。

云原生主线：

ECS/GPU、VPC、CLB、TOS、RDS/Redis、VKE、镜像仓库、微服务、DevOps、监控日志。

### 4. 需求调研九问

1. 业务目标是什么？
2. 谁使用，使用频率多高？
3. 当前系统和数据在哪里？
4. 最大痛点是什么？
5. 一期必须做什么？
6. 性能、并发、可用性要求是什么？
7. 安全、合规、部署形态要求是什么？
8. 预算、时间和决策链是什么？
9. POC 成功标准是什么？

### 5. 方案设计八块

1. 背景目标
2. 现状痛点
3. 总体架构
4. 产品组合
5. 数据流和调用链
6. 安全与高可用
7. POC 和实施计划
8. 成本收益与风险

### 6. POC 七要素

1. 目标
2. 范围
3. 数据
4. 环境
5. 指标
6. 分工
7. 报告和下一步

### 7. RAG 架构

文档接入 -> 清洗解析 -> 切片 -> 向量化 -> VikingDB/知识库 -> 检索召回 -> 重排 -> 火山方舟/豆包生成 -> 引用追溯 -> 权限审计 -> 反馈优化

### 8. 数据中台架构

数据源 -> 数据集成 -> 存储计算 -> 数据开发治理 -> 数据资产指标 -> 数据服务 -> BI/运营/风控/AI 应用

产品映射：

DataSail/DataLeap -> TOS/LAS/EMR/ByteHouse/Flink -> DataLeap 治理 -> DataWind/数据服务/AI 应用

### 9. 云原生迁移架构

应用评估 -> 镜像化 -> VKE 集群 -> 服务暴露 -> CI/CD -> 监控日志 -> 灰度发布 -> 弹性伸缩 -> 安全治理

### 10. 常见加分表达

- 我会先确认客户业务目标，而不是直接推荐产品。
- POC 必须有明确成功标准，否则很难推动采购。
- 大模型落地要关注数据、评测、权限、成本和人工兜底。
- 售前要促成项目，也要守住交付边界。
- 产品缺口要转成可评估的业务需求，而不是简单说客户想要。
- 技术方案要能被业务方、技术方和采购方同时理解。

### 11. 常见扣分表达

- “我觉得直接上大模型就行。”
- “这个功能应该都能做。”
- “价格贵可以打折。”
- “售前主要就是做 PPT。”
- “客户要什么我们就做什么。”
- “竞品不如我们。”

### 12. 反问问题

- 这个岗位目前主要覆盖哪些行业和产品方向？
- 团队对售前的技术深度和商机推进深度分别有什么期待？
- POC 通常由售前主导到什么程度？
- 新人前三个月最关键的目标是什么？
- 团队如何沉淀行业方案、标杆案例和可复用材料？

## 三、最后一天只背这段

我理解火山引擎售前解决方案岗位的核心，是把火山在 AI 云、数据中台、云原生、音视频和行业实践上的能力，转化为客户能理解、能验证、能采购、能交付的方案。我的工作方式会从客户业务目标出发，先做需求调研和痛点拆解，再设计分层架构和产品组合，通过 POC 明确验证指标，最后配合销售完成技术澄清、投标讲标和交付衔接。我会特别关注方案的业务价值、技术可行性、安全合规、成本收益和交付边界。
