# 云与 AI 零基础学习手册

适用目标：火山引擎售前解决方案岗位面试  
适用基础：云计算和 AI 技术基础较弱，从 0 开始补课  
学习重点：能听懂客户需求、能看懂方案架构、能用售前语言解释技术价值

## 一、先建立整体地图

云计算和 AI 不要分开学。现在企业做 AI，通常不是“买一个模型”就结束，而是需要一整套技术底座：

业务系统 -> 数据 -> 云基础设施 -> 数据平台 -> AI 模型 -> AI 应用 -> 安全与运维

你可以把它理解成一家餐厅：

- 云服务器：厨房里的灶台和锅。
- 存储：仓库和冰箱。
- 网络：送菜通道。
- 数据库：菜单、订单和会员档案。
- 数据平台：把每天的订单、库存、会员行为整理成经营报表。
- AI 模型：聪明的厨师或顾问。
- 知识库/RAG：厨师可以随时查菜谱和规则。
- 安全与运维：门禁、监控、消防和巡检。

售前要做的事情，就是根据客户想开一家什么样的餐厅，帮他设计厨房、仓库、动线、人员和管理制度，而不是只卖一口锅。

## 二、云计算基础

### 1. 什么是云计算

云计算就是把计算、存储、网络、数据库、安全等 IT 能力，像水电一样通过互联网或专线提供给企业使用。

传统方式：

企业自己买服务器、建机房、招运维、装系统、扩容、备份。

云计算方式：

企业按需购买云资源，用多少买多少，需要扩容时快速增加资源。

一句面试话术：

云计算的核心价值是把 IT 基础设施从重资产、自运维模式，变成弹性、按需、可托管的服务模式，从而提升交付效率、降低初期投入和运维复杂度。

### 2. IaaS、PaaS、SaaS

#### IaaS：基础设施即服务

提供最底层的资源。

例子：

- 云服务器
- GPU 云服务器
- 云硬盘
- 对象存储
- VPC 网络
- 负载均衡

类比：

租毛坯房，水电墙面都有，但你自己装修。

#### PaaS：平台即服务

提供开发和运行平台。

例子：

- 数据库服务
- Kubernetes 容器服务
- 大数据平台
- 机器学习平台
- 消息队列

类比：

租精装房，基础装修好了，你直接布置家具。

#### SaaS：软件即服务

直接使用软件。

例子：

- CRM
- BI 报表系统
- 在线协作文档
- 智能客服系统

类比：

住酒店，直接使用完整服务。

### 3. 公有云、私有云、混合云

#### 公有云

云厂商统一建设资源池，多个客户共享底层基础设施，但通过账号、网络、权限隔离。

适合：

- 快速上线
- 弹性需求明显
- 中小企业或创新业务
- 对成本敏感

#### 私有云

为一个客户单独建设和使用的云环境，可能部署在客户机房或专属环境。

适合：

- 金融、政企、医疗等强合规行业
- 数据不能出域
- 对控制权要求很高

#### 混合云

部分系统在本地，部分系统在公有云，两边通过专线、VPN、网关等连接。

适合：

- 老系统不能立刻迁移
- 核心系统本地部署，创新业务上云
- 灾备和弹性扩容

面试表达：

部署形态不是固定答案，要根据客户的数据安全、合规要求、系统现状、预算和上线周期选择。

## 三、云基础产品怎么理解

### 1. 云服务器 ECS

云服务器就是云上的虚拟电脑。企业可以在上面部署网站、后台服务、数据库客户端、业务程序等。

关键概念：

- CPU：计算能力。
- 内存：运行程序时临时存放数据的空间。
- 系统盘：安装操作系统。
- 数据盘：存放业务数据。
- 镜像：创建服务器的模板。
- 安全组：服务器的网络防火墙。

售前要会讲：

客户业务系统需要运行环境，云服务器提供弹性计算资源，可以按业务规模选择规格，并配合负载均衡和弹性伸缩应对流量变化。

### 2. GPU 云服务器

GPU 最早用于图形计算，现在广泛用于 AI 训练和推理。

CPU 和 GPU 区别：

- CPU 像少数几个能力很强的专家，适合复杂逻辑。
- GPU 像很多个同时工作的工人，适合大量重复计算。

AI 为什么需要 GPU：

大模型训练和推理涉及大量矩阵计算，GPU 更适合并行处理。

售前场景：

- 大模型训练
- 图片/视频生成
- 语音识别
- 推荐系统
- 离线推理
- 云游戏渲染

### 3. 对象存储 TOS

对象存储适合存放海量非结构化数据。

非结构化数据包括：

- 图片
- 视频
- 音频
- 文档
- 日志
- 模型文件
- 备份文件

类比：

对象存储像一个无限扩展的云端仓库，每个文件都有自己的地址。

售前话术：

对象存储适合做数据湖底座、内容分发源站、备份归档和 AI 多模态数据存储，优势是容量大、可靠性高、按量付费、便于与计算和数据产品集成。

### 4. 数据库

数据库是结构化数据的管理系统。

常见类型：

- MySQL/PostgreSQL：关系型数据库，适合订单、用户、交易等结构化数据。
- Redis：缓存数据库，适合高频读取、会话、排行榜。
- MongoDB：文档数据库，适合 JSON 文档和灵活结构。
- HBase：大规模宽表，适合海量明细数据。
- ByteHouse/ClickHouse 类：分析型数据库，适合报表和分析查询。

面试时不要只背名字，要会按场景选：

- 交易系统：MySQL/PostgreSQL。
- 高并发缓存：Redis。
- 日志和行为分析：大数据/分析型数据库。
- 海量图片视频文件：对象存储。

### 5. VPC 网络

VPC 是云上的私有网络。

你可以理解成客户在云上划出一个自己的园区，里面有不同楼栋和房间。

关键概念：

- VPC：大的网络空间。
- 子网：VPC 里的小网段。
- 路由表：决定流量往哪里走。
- 安全组：控制服务器端口能不能被访问。
- EIP：公网 IP。
- NAT 网关：让内网资源访问公网。
- VPN/专线：连接客户本地机房和云上网络。

售前常问：

客户系统要不要公网访问？是否要和本地机房打通？是否有内外网隔离？哪些端口需要开放？

### 6. 负载均衡

负载均衡把用户请求分发到多台服务器。

为什么需要：

- 单台服务器扛不住高并发。
- 某台服务器故障时，流量可以转到健康服务器。
- 支持横向扩容。

类比：

餐厅门口有一个领位员，把客人分配到不同桌子，避免某张桌子忙死、其他桌子空着。

### 7. CDN 和边缘

CDN 是内容分发网络，把图片、视频、网页等内容缓存到离用户更近的节点。

价值：

- 用户访问更快。
- 源站压力更小。
- 弱网体验更好。
- 支撑大流量分发。

边缘计算是在靠近用户的位置运行部分计算任务。

适合：

- 视频直播
- 云游戏
- 车联网
- IoT
- 低延迟互动

## 四、云原生基础

### 1. 什么是云原生

云原生是一套让应用更适合云环境的技术和方法，包括容器、Kubernetes、微服务、DevOps、弹性伸缩和可观测性。

一句话：

云原生的目标是让应用交付更快、扩展更灵活、故障恢复更容易、资源利用更高。

### 2. 容器 Docker

容器把应用和它依赖的环境打包在一起。

传统问题：

开发电脑能跑，测试环境不能跑，生产环境又出问题。

容器解决：

把代码、依赖、运行环境打成镜像，到哪里运行都尽量一致。

类比：

容器像标准集装箱。货物装进统一规格的箱子，火车、轮船、卡车都能搬。

### 3. Kubernetes

Kubernetes，简称 K8s，是管理容器的平台。

它负责：

- 把容器部署到合适的机器上。
- 容器挂了自动拉起。
- 根据流量扩容缩容。
- 做服务发现和负载均衡。
- 支持滚动发布和回滚。

核心概念：

- Pod：K8s 里最小运行单元，通常包含一个或多个容器。
- Deployment：声明我要运行几个副本，如何升级。
- Service：给一组 Pod 一个稳定访问入口。
- Ingress：管理外部流量进入集群。
- ConfigMap/Secret：管理配置和密钥。
- Namespace：做资源隔离。
- HPA：根据负载自动扩缩容。

售前话术：

客户从传统虚拟机部署转向 Kubernetes，核心收益是标准化应用交付、提升弹性伸缩能力、降低环境不一致问题，并通过灰度发布和可观测性提升稳定性。

### 4. 微服务

微服务是把一个大系统拆成多个小服务。

例子：

电商系统可以拆成：

- 用户服务
- 商品服务
- 订单服务
- 支付服务
- 库存服务
- 营销服务

好处：

- 每个服务可以独立开发、部署、扩容。
- 某个模块故障不一定拖垮全站。
- 团队分工更清晰。

难点：

- 服务之间调用复杂。
- 链路追踪更重要。
- 配置、权限、发布、监控都更复杂。

### 5. DevOps

DevOps 是开发和运维协同的方法。

关键词：

- CI：持续集成，代码提交后自动构建和测试。
- CD：持续交付/部署，自动发布到测试或生产环境。
- 灰度发布：先让少量用户使用新版本。
- 蓝绿发布：新旧两套环境切换。
- 回滚：出问题快速退回旧版本。

售前关注点：

客户要的不是 Jenkins 或流水线本身，而是更快上线、更少人为错误、更短故障恢复时间。

## 五、大数据基础

### 1. 为什么需要大数据

普通数据库适合处理交易类数据，但当数据量达到 TB、PB，且来源很多、格式复杂、分析需求多时，就需要大数据平台。

典型数据：

- 用户点击日志
- App 行为数据
- 交易流水
- IoT 设备数据
- 视频和图片元数据
- 推荐系统特征
- 客服工单

### 2. 数据仓库

数据仓库是面向分析的数据系统。

它通常把多个业务系统的数据汇总、清洗、建模，形成统一口径的数据。

典型问题：

销售部门说 GMV 是 100 万，财务说是 95 万，运营说是 105 万。为什么？因为口径不同。

数据仓库要解决：

- 数据统一
- 指标统一
- 分析高效
- 历史可追溯

### 3. 数据湖

数据湖可以存放原始数据，包括结构化、半结构化、非结构化数据。

数据仓库像整理好的图书馆，数据湖像容量很大的资料库。

AI 时代数据湖更重要，因为 AI 需要大量图片、视频、音频、文档等多模态数据。

### 4. 湖仓一体

湖仓一体是把数据湖的灵活和数据仓库的治理、性能结合起来。

面试话术：

湖仓一体适合既要低成本存储多类型数据，又要支持高性能分析、统一元数据和数据治理的企业场景。

### 5. 常见大数据组件

- Hadoop：早期大数据计算和存储生态。
- HDFS：分布式文件系统。
- Hive：用 SQL 查询大数据。
- Spark：批处理和大规模计算。
- Flink：实时流计算。
- Kafka：消息队列，常用于实时数据管道。
- Iceberg/Hudi：湖格式，支持数据湖表管理。
- StarRocks/Doris/ClickHouse：高性能分析查询。

### 6. 数据治理

数据治理解决“数据能不能被信任和复用”的问题。

包括：

- 元数据：数据的说明书。
- 血缘：数据从哪里来，经过哪些加工，到哪里去。
- 数据质量：是否为空、重复、异常、延迟。
- 指标口径：大家对同一个指标定义一致。
- 权限：谁能看哪些数据。
- 审计：谁在什么时候访问了什么数据。
- 成本治理：哪些任务和表浪费资源。

售前话术：

很多企业不是没有数据，而是数据不好找、不敢用、口径不一致、质量不可控。数据治理的价值是让数据资产可发现、可理解、可信任、可复用。

## 六、AI 基础

### 1. AI、机器学习、深度学习、大模型

AI 是总称，让机器表现出智能。

机器学习是 AI 的一种方法，让机器从数据中学习规律。

深度学习是机器学习的一类方法，使用神经网络处理复杂数据。

大模型是参数规模很大、训练数据很多、能力泛化更强的模型。

关系：

AI > 机器学习 > 深度学习 > 大模型

### 2. 训练和推理

训练：

用大量数据教模型学习规律。

推理：

模型学好后，用户输入问题，模型给出答案。

类比：

训练像学生备考学习，推理像考试答题。

售前场景：

- 客户要自研模型：关注训练平台、GPU、数据、调度、成本。
- 客户只调用模型：关注推理 API、延迟、并发、稳定性、价格。

### 3. Prompt

Prompt 是给大模型的指令。

例如：

请你扮演客服，根据以下产品说明回答用户问题。如果资料中没有答案，请回答“暂未查到相关信息”。

好的 Prompt 可以提升回答稳定性，但不能替代数据治理和评测。

### 4. Token

Token 是模型处理文本的基本单位，可以粗略理解为字或词片段。

为什么重要：

- 模型按 Token 计费。
- 输入越长、输出越长，成本越高。
- 上下文长度决定一次能放多少内容。

售前话术：

企业大模型应用要关注 Token 成本，可以通过模型选型、摘要、缓存、检索增强、限制输出长度等方式优化。

### 5. Embedding

Embedding 是把文本、图片等内容转成向量。

向量可以表达语义相似度。

例子：

“如何修改密码”和“忘记密码怎么办”字面不同，但语义接近，向量距离会比较近。

Embedding 是 RAG 和语义搜索的基础。

### 6. 向量数据库

向量数据库专门存储和检索向量。

用途：

- 知识库问答
- 语义搜索
- 推荐系统
- 相似图片检索
- 文档去重

在火山引擎语境里，可以关注 VikingDB。

### 7. RAG

RAG 是检索增强生成。

为什么需要 RAG：

大模型本身不一定知道企业内部知识，也可能编答案。RAG 先从企业知识库里检索相关资料，再让模型基于资料回答。

流程：

用户问题 -> 向量检索相关文档 -> 把文档片段放进 Prompt -> 大模型生成答案 -> 返回引用来源

优点：

- 降低幻觉。
- 可更新知识。
- 可追溯来源。
- 不一定需要重新训练模型。

局限：

- 文档质量差，答案也会差。
- 切片不合理会影响召回。
- 权限控制必须做好。
- 仍然需要评测和人工反馈。

### 8. Fine-tuning 精调

精调是用特定数据继续训练模型，让模型更适合某类任务或风格。

适合：

- 固定格式输出。
- 特定领域表达。
- 分类、抽取、审核等任务。

不一定适合：

- 频繁变化的知识问答。此时 RAG 更合适。

面试表达：

企业知识更新频繁时，优先考虑 RAG；如果是稳定任务模式和输出风格，可以考虑精调。实际方案中也可以 RAG 和精调结合。

### 9. Agent

Agent 是能调用工具、规划步骤、完成任务的大模型应用形态。

普通问答：

用户问，模型答。

Agent：

用户给目标，模型拆步骤，调用搜索、数据库、API、表格、工单系统等工具完成任务。

例子：

“帮我分析上周销售下降原因，并生成汇报。”

Agent 可能会：

1. 查询销售数据。
2. 对比历史趋势。
3. 查库存和活动数据。
4. 总结原因。
5. 生成 PPT 或报告。

售前注意：

Agent 更接近真实业务流程，但风险也更高，需要权限、审计、工具边界、人工确认和失败回滚。

## 七、AI 落地架构

### 1. 企业知识库问答架构

数据源：

- PDF
- Word
- 网页
- FAQ
- 工单
- 数据库

处理：

- 清洗
- 解析
- 切片
- 打标签
- 向量化

检索：

- 向量数据库
- 关键词检索
- 重排

生成：

- 大模型
- Prompt 模板
- 引用来源

治理：

- 权限
- 审计
- 敏感词
- 评测
- 反馈

一句话：

知识库问答不是简单上传文档，而是数据处理、语义检索、大模型生成和安全运营的组合工程。

### 2. 智能客服架构

客户入口：

- App
- 网站
- 小程序
- 电话/语音
- 企业微信

能力层：

- 意图识别
- 知识库问答
- 多轮对话
- 工单创建
- 人工转接
- 坐席辅助

数据层：

- 产品资料
- FAQ
- 历史工单
- 用户订单
- 服务记录

指标：

- 自助解决率
- 人工转接率
- 平均处理时长
- 用户满意度
- 答案准确率

### 3. 企业数据智能架构

目标：

让业务人员用自然语言查询数据、生成分析、发现异常。

架构：

业务数据库/日志 -> 数据集成 -> 数据仓库/湖仓 -> 指标平台 -> 语义层 -> 大模型 -> 数据问答/分析报告

难点：

- 指标口径必须统一。
- 权限必须清楚。
- 大模型生成 SQL 要校验。
- 重要结论要可追溯。

## 八、火山引擎相关产品怎么串起来

## 九、售前面试中技术问题怎么答

### 题目：你不懂某个技术怎么办？

参考回答：

如果客户或面试官问到我不熟悉的细节，我会先确认问题所属场景和目标，不会直接硬答。然后我会把自己确定的部分说清楚，比如这个能力在架构中解决什么问题、依赖哪些上下游、需要验证哪些指标。对于具体参数或产品限制，我会查官方文档或拉产品专家确认，再给客户明确回复。售前可以不知道所有细节，但必须保证表达准确、边界清楚、响应及时。

### 题目：企业做 AI 应用，为什么不能只调 API？

参考回答：

因为生产级 AI 应用除了模型 API，还需要企业数据、权限、安全、评测、成本控制、系统集成和运维。比如知识库问答需要文档解析、切片、向量检索、引用追溯、权限隔离和日志审计；智能客服需要人工转接、工单系统集成和满意度评估。所以 API 是能力入口，但真正落地要做端到端工程。

### 题目：RAG 和精调怎么选？

参考回答：

如果客户主要问题是企业知识问答，并且知识经常更新，我会优先选 RAG，因为它能快速接入最新知识，并支持引用追溯。如果客户需要模型稳定输出某种格式、语气或行业任务模式，比如分类、抽取、审核，可以考虑精调。实际项目里也可以先用 RAG 快速验证价值，再根据效果决定是否精调。

### 题目：云原生有什么价值？

参考回答：

云原生的价值不是“用了 Kubernetes”，而是提升应用交付和运行效率。它通过容器解决环境一致性，通过 Kubernetes 做自动调度、扩缩容和故障恢复，通过 DevOps 提升发布效率，通过监控和日志提升可观测性。对客户来说，最终体现为上线更快、扩容更快、故障恢复更快、资源利用率更高。

## 十一、面试前速背

### 1. 云计算

云计算是把计算、存储、网络、数据库等 IT 能力通过服务方式提供给企业，核心价值是弹性、按需、托管和快速交付。

### 2. 云原生

云原生通过容器、Kubernetes、微服务、DevOps 和可观测性，让应用更容易部署、扩展、恢复和运维。

### 3. 大数据

大数据平台解决海量、多源、多类型数据的采集、存储、计算、治理和分析问题，帮助企业形成统一数据资产和业务洞察。

### 4. AI 大模型

大模型提供通用理解、生成和推理能力，但企业落地需要结合数据、知识库、权限、安全、评测和业务系统集成。

### 5. RAG

RAG 是先检索企业知识，再让大模型基于知识生成答案，适合企业知识库问答，可以降低幻觉并支持来源追溯。

### 6. 售前技术表达

我会先确认客户业务目标和约束，再设计架构和产品组合，通过 POC 验证关键指标，最后明确成本收益、风险和交付边界。

## 十三、学习建议

不要试图一下子学成云架构师。你面试售前解决方案，第一阶段最重要的是：

1. 能听懂客户说的技术词。
2. 能知道每类产品解决什么问题。
3. 能按场景组织方案。
4. 能设计 POC 验证指标。
5. 能承认细节未知，但知道怎么查证和推进。

建议每天用 30 分钟做一件事：

选一个场景，写出“业务目标、痛点、架构、产品组合、POC 指标”。连续练 10 个场景，你的售前表达会提升很快。
