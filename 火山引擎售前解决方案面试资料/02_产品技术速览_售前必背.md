# 产品技术速览：售前必背

## 一、售前要掌握的产品地图

售前不需要像产品经理一样记住每个控制台按钮，但要能做到：

- 知道客户场景应该落在哪类产品组合。
- 能讲清楚产品在架构里的位置。
- 能说出关键能力、限制、依赖和验证指标。
- 能把产品能力翻译成业务价值。

## 二、AI 与大模型产品

### 1. 豆包大模型

豆包是字节跳动自研大模型。2026 年公开资料显示，豆包大模型 2.0 系列已面向企业和开发者上线 API，包含 Pro、Lite、Mini、Code 等定位不同的模型形态。

面试表达：

- Pro：适合复杂推理、Agent、高难度任务。
- Lite：强调能力和性价比平衡。
- Mini：速度和成本优先，适合高频轻量任务。
- Code：偏编程任务，可与开发工具结合。

售前视角要补充：

- 不能只按模型“最强”选型，要按任务复杂度、延迟、吞吐、预算、上下文长度、数据安全来选。
- 企业场景通常需要评测集、提示词模板、知识库、权限、日志审计和人工兜底。

### 2. 火山方舟

火山方舟是一站式大模型服务平台，提供模型推理、评测、精调等全流程服务，并搭载豆包及业界主流大模型。

典型场景：

- 企业智能客服
- 知识库问答
- 文档审核与摘要
- 营销内容生成
- 研发助手
- 金融研报辅助分析
- Agent 工作流

售前讲法：

客户要的不是“一个模型”，而是可运营、可评估、可集成、可审计的大模型应用底座。火山方舟可以承担模型接入、调用、评测、精调和安全互信等环节，降低企业从试点到生产的工程复杂度。

### 3. VikingDB、知识库、RAG

VikingDB 是火山引擎的向量数据库，面向海量向量数据的存储、索引和检索，支持智能问答、智能搜索、推荐、去重等场景。Viking 知识库基于向量检索提供文档管理、语义检索、智能对话等能力。

RAG 方案基本链路：

数据源 -> 文档解析 -> 切片 -> 向量化 -> 向量库存储 -> 检索召回 -> 重排 -> 大模型生成 -> 权限与审计 -> 反馈闭环

POC 指标：

- 召回准确率
- 答案忠实度
- 幻觉率
- 首字延迟和总响应时间
- 知识更新时效
- 权限隔离是否正确
- 成本/千次问答

### 4. 机器学习平台

火山引擎机器学习平台面向机器学习应用开发者，覆盖数据托管、代码开发、模型训练、模型部署等生命周期。

售前场景：

- 客户有自研模型团队，需要统一训练和推理平台。
- 客户 GPU 资源利用率低，训练任务排队严重。
- 客户希望把模型开发、训练、部署、监控标准化。
- 客户需要支持 PyTorch、TensorFlow 等多框架训练推理。

### 5. AI 数据湖 LAS

LAS 是面向多模态数据湖的服务，孵化于字节跳动大模型训练场景。它强调多模态元数据统一管理、AI 算子处理、权限控制，并可对接火山方舟，支撑机器学习、模型训练、模型微调等 AI 场景。

售前讲法：

大模型落地的瓶颈往往不在模型调用，而在企业数据是否可用、可管、可追溯、可合规。LAS 适合承接多模态数据沉淀、治理和 AI 数据准备。

## 三、数据中台与大数据产品

### 1. DataLeap

DataLeap 是一站式大数据中台解决方案，覆盖实时和离线数据集成、数据开发、智能运维、数据治理、资产管理等能力。

客户痛点：

- 多业务系统数据孤岛
- 数仓开发效率低
- 指标口径不一致
- 数据质量不可控
- 作业运维复杂
- 成本治理缺乏抓手

对应价值：

- 用数据集成和数据开发提升研发效率。
- 用数据资产和指标平台统一口径。
- 用数据质量、血缘、权限和审计提升治理能力。
- 用资源优化、智能运维降低成本和运维压力。

### 2. EMR

EMR 是云原生开源大数据平台，集成 Hadoop、Spark、Hive、Flink、Ray、Hudi、Iceberg、StarRocks 等生态组件，并强调开源兼容、弹性伸缩、存算分离、云原生运维。

售前常见场景：

- 自建 Hadoop 集群上云
- 离线数仓和交互式分析
- 实时分析
- AI 数据预处理
- 大模型离线推理
- 湖仓一体架构升级

### 3. ByteHouse、Flink、DataSail、DataWind

可按数据链路记忆：

- DataSail：数据集成，把源端数据同步到目标端。
- Flink：实时计算，处理低延迟流式任务。
- EMR/Spark/Hive：离线计算和大数据生态。
- ByteHouse：云原生数仓和分析。
- DataLeap：研发治理和统一工作台。
- DataWind：BI 和数据洞察。

## 四、云原生产品

### 1. VKE 容器服务

VKE 是火山引擎 Kubernetes 容器服务，面向容器化应用部署、管理、扩缩容和运维。

客户痛点：

- 自建 K8s 运维复杂。
- 应用发布慢，环境不一致。
- 微服务治理、弹性伸缩和资源利用率不足。
- AI 推理、批处理等异构负载需要更好的调度。

售前方案要点：

- 集群规划：地域、可用区、节点池、资源配额。
- 网络：VPC、子网、Pod 网络、安全组、Ingress、负载均衡。
- 存储：云盘、NAS、TOS、vePFS。
- 发布：蓝绿、灰度、滚动升级。
- 运维：监控、日志、告警、审计。
- 安全：镜像扫描、RBAC、NetworkPolicy、密钥管理。

### 2. 微服务与 DevOps

面试中常见技术点：

- Docker、Kubernetes、Helm、镜像仓库。
- Spring Cloud、Dubbo、Service Mesh。
- CI/CD、Jenkins、GitLab CI、Maven、制品库。
- 灰度发布、蓝绿发布、回滚、可观测性。

回答时要体现你知道“工具不是目的”。客户要的是交付速度、稳定性、弹性、成本和治理。

## 五、云基础设施

### 1. 计算

- 云服务器：承载通用业务应用。
- GPU 云服务器：AI 训练、推理、图形渲染、云游戏等。
- 弹性伸缩：应对流量波峰波谷。

### 2. 网络

- VPC：隔离网络环境。
- 子网和路由：规划业务分区。
- 负载均衡：高可用入口和流量分发。
- NAT、EIP、专线/VPN：公网访问和混合云连接。

### 3. 存储

- TOS：对象存储，适合海量非结构化数据、备份、内容分发、AI 数据底座。
- 文件存储：共享文件访问，适合容器、AI、传统应用迁移。
- 块存储：云服务器系统盘和数据盘。

### 4. 数据库

- RDS MySQL/PostgreSQL：传统关系型业务。
- Redis：缓存、会话、排行榜、热点数据。
- MongoDB/HBase：文档、宽表、大规模半结构化数据。
- ByteHouse：分析型场景。

## 六、视频云、CDN 与边缘

火山引擎在音视频和内容分发上有字节实践背景。售前可以关注：

- RTC：实时音视频、互动直播、在线教育、智能客服实时对话。
- 点播/直播/视频处理：内容平台、教育、泛娱乐。
- veImageX：图片处理、分发、画质优化。
- CDN 与边缘：低延迟、高并发分发，边缘计算、云游戏、车载娱乐等。

## 七、一个通用架构回答模板

如果面试官问“客户要做企业知识库问答，你怎么设计”，可以这样答：

1. 需求确认  
   明确知识来源、用户角色、权限体系、更新频率、准确率要求、并发量、部署形态和合规要求。

2. 数据层  
   文档、网页、数据库、工单、制度文件进入 TOS 或数据湖，做清洗、解析、切片、元数据标注。

3. 检索层  
   使用 VikingDB/知识库进行向量化、索引、召回和重排，结合关键词检索提升专业术语召回。

4. 模型层  
   通过火山方舟接入豆包或适合的模型，按任务选择 Pro/Lite/Mini 等。

5. 应用层  
   接入企业门户、客服系统、飞书/企微或业务系统，提供问答、摘要、生成、流程触发。

6. 治理层  
   做权限隔离、日志审计、敏感信息过滤、评测集、人工反馈和持续优化。

7. POC 指标  
   准确率、召回率、幻觉率、响应时延、并发、知识更新时效、单次调用成本、用户满意度。
